# 麻雀 点数計算 練習アプリ — タスク分解・実装計画（TASKS）

> `SPEC.md` に基づく実装計画。各フェーズは依存関係・完了条件（DoD）・リスクを明記。
> **最大リスクは P2 点数計算エンジンの正確性**。ここを厚いテストで固めてからUIを載せる。

## 進め方の原則

- エンジンは**純粋関数＋型安全**（`src/engine`）。UIから独立してテスト可能に保つ。
- **テストファースト寄り**: P2〜P5 はユニットテストと同時に実装。問題バンク（P3）を回帰テストの正解データに使う。
- 依存の順序: **P0 → P1 → P2 → P3 →（P4/P5）→（P6/P7/P8）→ P9 → P10**。P6/P7/P8、P4/P5 は一部並行可。

## P2. 点数計算エンジン（最重要）

**目的**: `SPEC.md` §5 を厳密に実装する。

- **和了分解**: 手牌を「4面子1雀頭」「七対子」「国士無双」の各形に分解（複数解を列挙）
- **役判定**: 各分解に対し、条件（門前/鳴き・ツモ/ロン・リーチ・自風/場風）を加味して成立役を判定（喰い下がり含む）
- **符計算**: 副底・門前ロン・ツモ・待ち・面子・雀頭、平和20符・七対子25符・喰い平和30符切り上げ、10符切り上げ
- **翻計算**: 役の翻＋ドラ＋裏ドラ＋赤ドラ
- **点数計算**: 基本点→親子×ツモ/ロンの支払い、満貫〜役満・数え役満、100点切り上げ
- **高点法**: 全解釈のうち最終点数が最大のものを選択
- 出力は `ScoreResult`（役一覧・翻・符・支払い・区分・別解メモ）

**DoD**: §5 の各ルールを網羅するユニットテストが緑。代表点数（30符4翻=子7700 等）と満貫以上、役満、平和ツモ20符、七対子25符、喰い下がり、赤/裏ドラ、高点法の別解ケースを検証。
**リスク**: **高**。和了分解の網羅と高点法が難所。テストを厚く。

## P3. 問題モデル & 検証済みバンク

**目的**: 正確性の担保と初期コンテンツ。

- `Problem` の JSON スキーマ確定（`SPEC.md` §6）
- **検証済みシード問題 50〜100 問**を作成（門前/鳴き/七対子/役満/赤・裏ドラ/高点法を含む多様性）
- バンクの各問題を**エンジンの回帰テストフィクスチャ**として使用（バンクの `answer` とエンジン出力が一致すること）
- 問題に `tags`（`fuType` / `yakuCategories`）を付与
- **バンク再生成**（P11・ドラ必須化に伴う）: 修正後のジェネレータでバンクを作り直し、**全問がドラ表示牌1枚以上**を満たすことをフィクスチャで検証（§5.4）

**DoD**: バンク全問でエンジン出力＝記載の正解。タグが付与済み。**全問がドラ表示牌1枚以上**。
**リスク**: 中（作問の手間と正確性）。エンジンとの二重チェックで担保。

## P4. 問題ジェネレータ

**目的**: ランダム出題の主生成。

- ランダムな和了形（手牌・副露・上がり牌・条件・ドラ）を生成
- **妥当性検証**: エンジンで採点し、**最低1役成立**する手のみ採用（ドラのみ等は破棄）
- 局条件は場風=東/南、自風=東南西北の範囲でランダム
- 生成問題にも `tags` を付与
- **ドラ表示牌の枚数は「1＋槓の数」**（実戦準拠。§4.1 / §5.4）。リーチ時は裏ドラ表示牌も同数

**DoD**: 生成→採点可能な正当問題が安定して得られる（無効手のリトライ含む）。分布が偏りすぎない。**生成問題は必ずドラ表示牌を1枚以上含む**。
**リスク**: 中（無効手の多発による生成効率）。

**DoD**: どの問題でも妥当な選択肢集合が生成され、正解が必ず1つ含まれる。
**リスク**: 低〜中（誤答が正解と重複/非現実値にならない整形）。

## P11. 符計算モード

**目的**: 最終点数に加え「**符**」を答えるモードを追加する（`SPEC.md` §4.0 / §5.6）。既存の最終点数モードは無改変で維持する。

- **ドラ表示牌の必須化**: ジェネレータを「1＋槓の数」枚生成に修正し（P4）、問題バンクを作り直す（P3）
- **共通化**: 七対子25符の `FuBreakdown` 生成を `engine/fu.ts` に関数化（`scoreHand` からも符モードからも参照）
- **符ディストラクタ**: `generateFuChoices(fuDetail, rng)` を新設（切り上げ忘れ=`subtotal`・±10符・待ち/ツモ符の付け忘れ=±2・25固定の混同。`SPEC.md` §7）
- **出題の絞り込み**: 符が点数に影響しない**満貫以上（`rank` 付き）を出題対象外**にする（`nextProblem` に `excludeMangan` オプション）
- **共通部品の抽出**: 局条件・ドラ表示・4択グリッド・符内訳表示を最終点数モードと共有する部品に切り出す
- **符モード画面**（`/fu/quiz`・`/fu/result`）: 手牌等の表示は流用、選択肢を符の4択に、解説は既存の符内訳（§4.4）を主役に流用
- **モード選択導線**: ホームに「最終点数」「符計算」の2導線を追加
- **成績**: `recordAnswer` をモード共通で利用（符モードは `byFuType` が有効）
- **テスト**: 符計算の代表ケース（平和ツモ20・七対子25・暗槓・単騎・カンチャン＋ツモ・喰い平和30切り上げ）、`generateFuChoices`、既存の最終点数モードにデグレなし

**DoD**: 符モードで出題→符回答→採点→符内訳付き解説→成績が通しで動作。満貫以上が出題されない。全出題でドラ表示牌が1枚以上表示される。既存の最終点数モードに回帰なし。
**リスク**: 中（既存UI/ストアの分岐、符版誤答の妥当性、バンク再生成の網羅性）。

## P14. 切り上げ満貫の採点反映＋「満貫切上」条件タグ表示

**目的**: 切り上げ満貫設定（P13）が**有効な場合**、**点数計算モード（`/quiz`）の採点を切り上げ満貫ルールで行い**（正解点数・4択・解説の点数をすべて切り上げ後に揃える）、同時に問題画面上部の局条件タグリスト（`QuizConditions`）の**一番左**に「満貫切上」タグを表示する。タグと実際の点数が常に整合する状態（前回検討の案B）で実装する。

> 方針決定（2026-07-08）: **① 設定ON時は採点も切り上げ満貫で行い、タグと点数を整合させる（案B）／② `QuizConditions` は表示専用を維持し `roundUpMangan?: boolean` prop で受け取る／③ 対象は点数計算モード（`/quiz`）のみ。符計算モードは満貫以上を出題せず切り上げ満貫は無関係のため、採点反映もタグも行わない（`FuQuizPage`・`nextProblem`・問題生成は不変）／④ タグ位置は windset バッジより前（先頭）。**

### 設計の要点（採点反映の仕組み）

- 問題の `answer: ScoreResult` は**標準ルール（`roundUpMangan=false`）で事前計算**され、バンク／生成に埋まっている。設定ON時はこれを**表示層（`QuizPage`）で切り上げ後の値に差し替える**。生成・バンク・`nextProblem` は標準ルールのまま保つ（符モードや出題分布の前提を壊さない）。
- **高点法の勝ち解釈は切り上げ満貫で不変**（切り上げは基本点1920帯の点数を引き上げるだけで、解釈間の順位を反転させ得ない）。したがって差し替えは `answer` の再スコアで安全に行え、`han/fu/yaku/fuDetail` は一致し、**`payment`・`rank`・`interpretationNote`（別解の点差）だけ**が切り上げ後の値に変わる。
- 差し替えは `scoreHand` に採点オプションを渡して**再スコア**する方式を採る（`payment`・`rank` に加え別解メモの点差まで一貫して更新できるため）。

## P17. ヘッダーの帯スタイル統一（トップ以外を `.home-topbar` に揃える）

トップ以外の全画面のヘッダーを、トップ（`.home-topbar`）と**同じ全幅の帯スタイル**に統一する（SPEC §8.3 に反映済み）。現状トップ以外は「浮遊カード」型（全周2px罫線＋角丸＋グロー影・`top:8px`・中央寄せカラム内で左右に余白）で、トップの「画面端まで伸びるクリーム帯＋下罫線のみ・角丸/影なし・`sticky top:0`」と見た目が食い違っている。**帯の見た目のみ**を揃え、各ヘッダー内のボタン・ナビ（ホーム/成績ピル）と見出し（h1）は**現状の配置・スタイルのまま**無改変とする。

### スコープ

**含む**: 各 scoped `.page-header` の**コンテナ装飾**（背景・罫線・角丸・影・sticky位置・全幅化）をトップ帯に合わせて差し替える。

**含まない**:
- `.page-header-link` / `.page-header-link-item`（ホーム/成績ピル＝ボタン・ナビ）と `.page-header h1`（見出しテキスト）のスタイル・マークアップ。
- トップのブランドマーク（牌チップ＋アプリ名）を他画面へ移植すること。
- ロジック・DOM構造・エンジン・出題・成績。

> **方針決定（2026-07-13）**: **① 実装は純CSSのみ**（各ページの TSX / DOM・`<PageHeader>` コンポーネントは無改変）。full-bleed は「中央寄せカラム内から画面端まで抜く」定番トリックで実現する（`margin-inline: calc(50% - 50vw)` で全幅化 → `padding-inline: max(16px, calc(50vw - 320px))` で内側テキストを 640px 列の端に合わせる → `margin-top: calc(-1 * var(--space-4))` で `.page-shell` 上パディングを相殺し帯を最上部へ）。**② 帯スタイルは各 scoped ブロックへ個別に適用**（本プロジェクトの「Flip7 トークンはページscopeに閉じ、他画面へ波及させない」方針を踏襲。`--fl-*` はヘッダーがそのページscopeの子孫のままなので参照可）。**③ ボタン・ナビ・見出しは触らない**。

**適用する帯スタイル（トップ `.home-topbar` と同値）**:
```css
/* コンテナのみ差し替え。h1 / -link / -link-item はそのまま */
position: sticky; top: 0; z-index: 10;
display: flex; align-items: center; justify-content: space-between; gap: 12px;
background: rgba(255, 248, 231, 0.86);
backdrop-filter: blur(10px); -webkit-backdrop-filter: blur(10px);
border-bottom: 2px solid rgba(43, 168, 162, 0.22);
/* 除去: 全周 border / border-radius / box-shadow / top:8px */
/* 全幅化（中央寄せカラムから抜く） */
margin-inline: calc(50% - 50vw);
margin-top: calc(-1 * var(--space-4));   /* .page-shell の上パディング相殺 */
padding: 10px max(16px, calc(50vw - 320px));  /* 内側テキストを 640px 列端へ */
```
