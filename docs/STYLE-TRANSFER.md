# STYLE-TRANSFER.md — CSS→Tailwind移行ルール

> 本書は、既存のプレーンCSS（CSSカスタムプロパティ＋ページスコープの子孫セレクタ）を採用している画面をTailwind CSSへ移行する際の**汎用ルールの正典**。T-014（点数計算モード）・T-015（ホーム画面）の実施で確定した判断を一般化したもの。個々のタスクの適用結果（対象ファイル名・具体的なクラス名の列挙・受け入れ基準の文面等）は `TASKS.md` の各タスク、要点は `SPEC.md` §8.3.1/§8.3.2 を参照。以降の画面（符計算・符分解・点数換算等）を移行する際は、まず本書のルールに従う。

---

## R1. Tailwind CSS v4・Preflight非導入

CSS-first設定（`tailwind.config.js`は使わない）で導入する。`tailwindcss/theme`・`tailwindcss/utilities`のみを読み込み、`tailwindcss/preflight`は読み込まない。

**理由**: Preflightは全要素へのグローバルリセットを伴うため、導入すると移行対象外の画面の見た目にも影響してしまう。「対象外画面は一切変更しない」という段階移行の前提と矛盾する。

## R2. `@layer theme, base, utilities;` を明示し、グローバル要素セレクタは `base` に置く

Tailwind導入用CSS（`src/styles/tailwind.css`）の先頭で `@layer theme, base, utilities;` とレイヤー順序を宣言する。素のHTML要素セレクタ（`a`・`button` 等、クラスを介さない全体スタイル）はこの `base` レイヤーの中に置く。

**理由**: CSSカスケードレイヤーは「非レイヤーのルールは詳細度に関係なく、どのレイヤーのルールよりも常に勝つ」という仕様を持つ。`index.css` のようなプレーンCSS側に `a { color: ... }` の要素セレクタを非レイヤーのまま残すと、Tailwindの `layer(utilities)` 内にあるどれだけ詳細度の高いユーティリティクラス（`text-fl-teal-dark` 等）を `<Link>`/`<a>` に直接付けても勝てず、意図しない色が出続ける。`base` に入れて `utilities` より前に順序づけることで、通常のカスケード（詳細度＋レイヤー順）に戻す。

## R3. 新規トークンの `@theme` 昇格は「色のみ」

新しく必要になったCSSカスタムプロパティのうち、**色**（既存の `--color-fl-*` パレットの一員となるもの）は `src/styles/tailwind.css` の `@theme` へ `--color-*` として追加し、`bg-*`/`text-*`/`border-*` 等のネイティブユーティリティとして使えるようにする。**形状（半径）・グロー影・モーション（イージング/所要時間）のトークンは `@theme` へ昇格させず**、移行対象ページの `.xxx-page` スコープのCSSにローカル残置し、Tailwindの任意値記法（`shadow-[var(--fl-glow-teal)]` 等）から参照する。

**理由**: 色はTailwindの命名規約（`--color-*`）と一致し、ネイティブユーティリティ化する価値が大きい。半径・グロー影・所要時間はTailwindの命名規約と直接対応せず、他ページと共有されない一回性の値であることが多く、`@theme` 化しても二重管理の解消効果が薄い。

## R4. 残置CSSパターン：構造は移す、共有・非移設物だけ残す

移行対象コンポーネントの構造・レイアウト（`flex`/`grid`/`padding`/文字サイズ等）はすべてTailwindユーティリティに置き換えて元のCSSファイルから削除する。元のCSSファイルに残してよいのは次の2種類のみ:

1. 他の未移行ファイル・共有コンポーネントとクラス名を共有しているCSSルール（R5参照）。
2. `@theme` へ昇格しない色以外のトークン（R3）と、それを使う `@keyframes`・ページ全体の背景等、Tailwindの1クラスで表現しづらい宣言。

## R5. 対象範囲の判定ルール

1. 移行対象コンポーネントが**自身の直下で描画する要素**（＝共有コンポーネントの内部出力ではない要素）のみをTailwindユーティリティに置き換える。
2. 移行対象コンポーネントが使うクラス名のうち、**他の未移行ファイルや共有コンポーネントと共有されているもの**は、既存のCSSルールを削除しない。移行対象ファイル側はそのクラス名の使用をやめてTailwindユーティリティで同じ見た目を再現するが、CSSルール自体はそのクラス名を使い続けるファイルのために残す（未使用化するだけで削除しない）。
3. 共有コンポーネント自体（`ChoiceGrid`・`QuizConditions`・`SidebarPageHeader` 等）のコード・マークアップ・クラス名は一切変更しない。
4. 移行対象コンポーネントが他ファイルと**一切クラス名を共有していない**場合（例: T-015のホーム画面）は、上記2の例外は生じず、構造・レイアウトのCSSを全面的に削除できる。

## R6. 擬似要素は実要素に変換する

装飾用の `::before`/`::after`（アクセントバー・リボンの帯など）は、`before:`/`after:` のTailwind任意値チェーンに変換せず、`aria-hidden="true"` 付きの実要素（`<span>` 等）に置き換えてTailwindユーティリティで直接スタイリングする。

**理由**: 多属性の擬似要素を `before:[...]` 任意値チェーンで表現すると可読性が著しく落ちる。実要素に変換すれば通常のTailwindユーティリティで完結し、CSSファイルへの残置も不要になる。

## R7. ブレークポイントは共通のものだけ `@theme` に昇格する

サイト全体で使う境界（例: 640px、既存 `--bp-sm-up`）は `@theme` の `--breakpoint-*` に昇格し、標準の `sm:` プレフィクスを使う。**特定ページでしか使わない一回限りの境界**（例: T-015の480px 2カラム切替）は `@theme` に追加せず、`min-[480px]:` のようなTailwindの任意値ブレークポイントバリアントをJSX側に直接書く。使われなくなった `@custom-media` 定義は `src/styles/breakpoints.css` から削除する。

## R8. テスト方針

- 既存のテストがクラス名セレクタに依存している場合は、`role`／表示テキスト／`data-testid` ベースのクエリに書き換える（自然な `role` やテキストがない要素にのみ `data-testid` を付与）。
- 元々テストがなかった画面（純粋な見た目の変更のみ）に、この移行のためだけに新規テストを追加する必要はない。受け入れ確認はブラウザ目視で行う。

## R9. 受け入れ基準の定型

- `npm install` 後、対象画面の見た目が移行前とピクセルレベルで変わらない（配色・余白・角丸・アニメーション含む）。スマホ幅（375px程度）・PC幅の両方で確認する。
- インタラクティブ要素の `hover`/`active` 状態が移行前と一致する。
- `prefers-reduced-motion` 設定時、登場アニメーション・hover移動が無効化される（移行前と同じ挙動）。Tailwindの `motion-reduce:` バリアントで実現する。
- 独自ブレークポイントを使っている場合はその境界前後の挙動が一致する。
- 移行対象外の画面の見た目に変化がない（共有クラスの温存漏れ・新規昇格トークンの副作用がないこと）。
- `npm test` / `npm run lint` が通る。
