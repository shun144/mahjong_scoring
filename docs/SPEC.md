# 麻雀 点数計算 練習アプリ — 仕様書（SPEC）

> 本書はアプリの唯一の正典（source of truth）。実装・タスクは本書に従う。ドキュメントは日本語で維持する。
> ただし**フォルダ構成・レイヤー分割・依存ルール**（bulletproof-react＋オニオンアーキテクチャによる構成）は本書ではなく **`ARCHITECTURE.md`** を正典とする。CSS→Tailwind移行のルールが `STYLE-TRANSFER.md` に切り出されているのと同じ扱い。

## 1. 目的・対象ユーザー・ゴール

- **目的**: 麻雀の点数計算を、スマホWebで反復練習できるドリルアプリを提供する。
- **対象ユーザー**: 点数計算を覚えたい初〜中級者。実戦で「何点か即答できる」状態を目指す人。
- **ゴール**:
  - ランダム出題された和了形を見て、正しい**最終点数**を素早く選べるようになる。
  - 間違えたときに**なぜその点数か**（役・翻・符・計算式）を理解できる。
  - **苦手分野**（符の型・役の種類）を可視化し、重点的に復習できる。

## 2. スコープ

### 2.1 含む

- 四人麻雀・標準リーチ麻雀の和了点計算
- 門前手／鳴き手（副露・喰い下がり）
- 七対子・特殊符（平和20符・七対子25符 等）
- 役満・数え役満
- 赤ドラ・ドラ・裏ドラ
- リーチ有無、門前ツモ、自風・場風・親子の条件
- 高点法（複数解釈時は最高点を正解）

### 2.2 含まない（スコープ外）

- **状況役**: 一発・海底摸月・河底撈魚・嶺上開花・槍槓・ダブルリーチ・天和・地和・人和・流し満貫
- **ハウス/ローカルルール**: ダブル役満・役満の複合・青天井
  - **切り上げ満貫**は例外。§4.8 の**設定画面で任意にON/OFFできる**任意ルールとして提供する（**既定ON**）。**点数計算モード**では設定に応じて採点に反映される（符計算モードは満貫以上を出題しないため対象外）。
- 三人麻雀（サンマ）、両対応
- 持ち点・ウマ・オカ・順位・供託（リーチ棒/本場）による加算
- タイマー／タイムアタック
- **オフライン対応**（Service Worker・オフラインキャッシュ）。ネットワーク不通でも動作するアプリ化はしない。
- 多言語対応（日本語のみ）

> 注: 状況役は手牌から推測不能なため除外。門前ツモとリーチは条件として与えるため含む。
> 注: **ファビコン・apple-touch-icon・Web アプリマニフェスト（§8.4）は含む**。マニフェスト（`site.webmanifest`・`display:standalone`）により**ホーム画面へのインストール（スタンドアロン表示）は可能**にするが、**Service Worker を持たないためオフラインでは動作しない**（＝完全な PWA ではなく、アイコン・名称・インストール体験のみを提供するマニフェスト）。

## 3. 用語集

| 用語                     | 意味                                                                                                 |
| ------------------------ | ---------------------------------------------------------------------------------------------------- |
| 翻（ハン）               | 役やドラの合計値。点数の指数的な係数。                                                               |
| 符（フ）                 | 手の形から算出する加算値。10符単位に切り上げる。                                                     |
| 親／子                   | 親=場に対する起家（自風が東の家）。ロン・ツモの支払いと点数が異なる。                                |
| ツモ／ロン               | 自摸和了／他家からの栄和。符と支払い形式が変わる。                                                   |
| 面前（門前）             | 鳴いていない状態。門前でのみ成立する役がある（リーチ・平和・七対子等）。                             |
| 鳴き（副露）             | ポン・チー・明槓で他家の牌を晒すこと。喰い下がりが発生する。                                         |
| 喰い下がり               | 鳴くと翻が下がる役（三色2→1 等）。                                                                   |
| ドラ／裏ドラ／赤ドラ     | 翻を増やすボーナス牌。**役ではない**（ドラのみでは和了不可）。                                       |
| ドラ表示牌／裏ドラ表示牌 | 実際のドラ／裏ドラを示す牌（次の牌がドラになる）。プレイヤーは表示牌からドラを読み替える必要がある。 |
| 高点法                   | 1つの手が複数解釈できるとき、**最も高い点数**になる解釈を正とするルール。                            |
| 数え役満                 | 13翻以上を役満（子32000/親48000）として扱う。                                                        |

## 4. 機能要件

### 4.0 出題モード

- **最終点数モード**（既定）: §4.2 の通り、和了形の**最終点数**を4択で答える。
- **符計算モード**: 同じ和了形に対し**符**を4択で答える（§5.6）。符が点数に影響しない**満貫以上（`rank` 付き）は符計算モードの出題対象外**とする。
- **符分解モード**: 同じ和了形に対し、符を**要素ごと**（上がり方・面子の符・雀頭・待ち／固定符手は固定符）に選んで組み立てる（§4.10）。符計算モードと同じく満貫以上は出題せず、符に無関係なドラ表示牌は出さない。
- **点数換算モード**: 手牌を出さず、**符・翻・親子・ツモ/ロン**の条件のみを提示し、対応する**最終点数**を4択で答える（§4.9）。和了形（手牌・ドラ・役）・高点法・問題バンクは用いず、`calculatePayment`（§5.3）による純粋な換算のみを扱う。満貫以上（符が点数に影響しない区分）は出題しない。
- モードは**ホーム画面で選択**する。最終点数／符計算／符分解は専用画面（`/quiz`・`/fu/quiz`・`/fu/parts` 等）を持ち、手牌・局条件・符内訳などの表示部品を共有する（符計算／符分解はドラ表示を省く）。点数換算は手牌を持たないため独立画面（`/convert`）で完結する（§4.9）。
- 成績・苦手復習（§4.5）は最終点数／符計算モードの回答を既存の集計（`byFuType`・`byYakuCategory`）に集約する。**符分解モード・点数換算モードは成績に連携しない**（符分解＝§4.10／点数換算＝§4.9）。

### 4.1 出題

- 問題は**ランダム**に提示する（ジェネレータ生成75%＋バンクからのシャッフル25%。生成失敗時はバンクにフォールバック）。
- **出題傾向の調整**: 苦手復習の重み（§4.5）に、問題の内容だけで決まるカテゴリ係数を乗じて頻度を微調整する（成績には依らない）。体感で出過ぎる**七対子・役満は抑制**し、薄い**満貫以上かつ50符以上の高符高翻手は増やす**。区分は排他（役満→七対子→満貫以上&50符+ の順に判定）。なお**符分解モードは七対子をさらに強く抑制**し実出題で約3%にする（七対子抑制係数をモード別に上書き。§4.10）。
- 出題画面に表示する情報:
  - **手牌**（純手牌＋副露があれば副露を横向きで表示）
  - **上がり牌**（ツモ牌／ロン牌）
  - **ツモ／ロンの別**
  - **ドラ表示牌**（実戦準拠。プレイヤーが表示牌から実際のドラを読み替える）
  - **裏ドラ表示牌**（リーチ時のみ提示。非リーチ時は非表示）
  - **自風**（東南西北）、**場風**（東 or 南）
  - **親／子**
  - **リーチ有無**
- **リーチ表示の位置（最終点数モード・符計算モード）**: 最終点数モード・符計算モードでは、リーチの有無を**上段の局条件バッジ列には出さず**、**アガリ牌（ツモ/ロンの色付き枠）の右隣**に表示する（横の並びは `ツモ/ロン → リーチ → ドラ表示牌` の順。符計算モードはドラ表示牌自体を出さないため `ツモ/ロン → リーチ` の並びになる）。表示は「リーチ」ラベル＋1000点棒の画像（`src/assets/tenbo/tenbo-1000.png`）を、アガリ牌・ドラ表示牌と同じ「ラベル上・中身下」の構成で示す。**非リーチの問題でも同じ幅の枠を確保して空表示**とし、リーチ有無でドラ表示牌の位置が動かないようにする。**符分解モード（§4.10）はリーチ自体を表示しない**（符に無関係なため元々非表示。表示方針の変更対象外）。**点数換算モード（§4.9）はリーチを条件に含まない**ため対象外。
- **リーチ棒画像サイズ（狭幅対応）**: 表ドラ＋裏ドラの合計枚数が多いケース（例: 表3枚＋裏3枚＝6枚）では `dora-indicator-group` の幅が広がり、`quiz-tile-header` の折り返し（`flex-wrap: wrap`）によってドラ表示ブロックごと次の行に落ちてしまうことがある。これを緩和するため、**狭幅（画面幅640px未満）ではリーチ棒画像（`.riichi-indicator-stick`）の幅をPC時の92pxから76px程度（約-17%）に縮小**する。PC幅（640px以上）は92pxのまま変更しない。この変更はドラ表示・リーチ表示の gap や padding には手を加えず、リーチ棒画像の幅のみを対象とする。**既知の制限**: 極端に狭い画面幅（360px程度以下）では、6枚のケースでもなお折り返しが解消しきらない場合があるが、今回のスコープでは許容する（追加対応は別課題）。リーチ棒は `QuizTileHeader` の共有スタイルのため、最終点数モード・符計算モードの両方に同様に適用される。
- **局条件の範囲**: 場風は東・南のみ。自風は東南西北。西場・北場は出題しない。
- **親子の一貫性**: **自風=東の家は必ず親**（東家＝親）、自風≠東なら必ず子。`自風=東 ⟺ 親` の双条件を常に満たす（「自風:東 / 子」のような不整合な条件は出題しない）。
- すべての出題は**最低1役以上が成立**する正当な和了形であること（ドラのみの手は出さない）。
- **自風牌雀頭の出題を意図的に確保する**: 符計算で頻出かつ間違えやすい「雀頭＝自風牌（役牌雀頭 +2符）」の形（例: 東場西家で雀頭が西）を、生成時に一定割合で明示的に作る。純ランダムでは自風牌が雀頭になる確率が低く（約 1/34）ドリルでほぼ練習できないため、標準形（4面子1雀頭）の生成で一定確率（既定 約 1/15）で雀頭を自風牌に固定する。自風牌が雀頭になると平和・断幺九が崩れるため、他に役が成立しない手は §4.1 冒頭の「最低1役」制約によりリトライで破棄される（結果として自風牌雀頭の出題は役牌の刻子・混一色・対々和・チャンタ等を伴いやすい）。**場風=自風のとき（東場東家）は連風牌の雀頭**となるが、標準ルールに従い連風牌雀頭も +2符 として扱う（§5.1「補足」の連風牌注記に準拠）。七対子には雀頭の概念がないため対象外。**エンジンの符計算（§5.2 の雀頭 +2符）は変更せず、生成頻度のみを調整する**。
- **牌の枚数制約**: 同一牌は**最大4枚**。麻雀の牌山は各牌4枚しかないため、5枚以上になる出題はしない。この4枚には**手牌（純手牌＋副露＋上がり牌）に加え、ドラ表示牌・裏ドラ表示牌も含める**（表示牌も牌山から取る実牌のため。§5.4）。赤ドラ違いは同一牌として数える。**槓（明槓・暗槓）は該当牌の4枚すべてを占有する**ので、同じ牌を他の面子（順子・刻子・雀頭）や上がり牌、さらにドラ／裏ドラ表示牌に重複して使うことはできない。
- **ドラ表示牌の枚数は「1 ＋ 手牌中の槓の数」**（基本1枚＋槓ごとに1枚。実戦準拠。§5.4）。必ず1枚以上あり、槓が無ければ1枚、四槓子なら5枚。裏ドラ表示牌はリーチ時のみ提示し、その場合も同じ枚数（1＋槓の数）を出す。
- **四槓子はリーチ無し**: 槓が4つの手（四槓子）は必ずリーチを付けない（四暗槓で門前であっても付けない）。

### 4.2 回答UI

- **最終点数モード**では回答は**最終点数**を選択肢ボタンから選ぶ。
- **選択肢の形式は親/子・ツモ/ロンで変わる**:
  - ロン: 単一値（例: `8000`）
  - 子ツモ: `X / Y`（左＝子の支払い、右＝親の支払い。子/親のラベルは付けない。例: `2000 / 4000`）
  - 親ツモ: `X オール`（例: `4000 オール`）
- **符計算モード**では回答は**符**（例: `30符 / 40符 / 50符 / 70符`）を選択肢ボタンから選ぶ（§5.6・§7）。
- 選択肢は**正解＋紛らわしい誤答の4択**（§7の方針で生成）を4つのボタンとして並べる。
- ボタンを押すと即座に採点し、正誤にかかわらず解説画面へ遷移する（別途の「回答する」ボタンは持たない）。
- **「次の問題へ」ボタンの見た目統一（最終点数モード）**: 回答後にインライン表示される「次の問題へ」ボタンは、未回答時のスキップボタン（`quiz-skip`／ティール枠のアウトラインピル・中央寄せ）と同一の見た目にする。回答の有無によってボタンのスタイル（色・形状・配置）が変わらないようにする。
- **「もう一度」ボタン（最終点数モード）**: 回答後、同じ問題を回答・採点状態だけリセットして解き直せる「もう一度」ボタンを、「次の問題へ」ボタンと**並べて**（同一の見た目・対等な重み。順序は「もう一度」→「次の問題へ」）表示する。符分解モード（`FuPartsQuizPage`）の「もう一度」（採点後に同じ問題を解き直す機能）と同じ考え方を最終点数モードにも導入する。**再回答は成績（今日の回答数・正答率・苦手タグ）に記録しない**（既存の「問題に戻る」復習と同じ二重計上防止の仕組みを使う）。「もう一度」は未回答時には表示しない（回答前は解き直す対象が無いため）。
  - 補足: 本アプリの差別化はテンポ＝「ワンタップ次へ」（`docs/OBJECTIVE.md`）だが、本ボタンは意図的にこの原則に例外を設ける判断とする。ユーザー確認の上で、正誤表示直後の主導線を「次の問題へ」単独から「もう一度／次の問題へ」の対等な2択に変更する。

### 4.3 採点

- **高点法**で正解を決定（複数解釈のうち最高点）。
- **最終点数モード**: 選択された点数が正解の支払い（形式含む）と一致すれば正答。
- **符計算モード**: 選択された符が正解の符（`ScoreResult.fu`、高点法適用後・丸め後）と一致すれば正答（§5.6）。

### 4.4 解説

- 正誤（○×）と正しい点数を示す（表示位置・見た目はモードにより異なる。下記参照）。
- **点数計算**（`QuizPage`のインライン結果・`ResultPage`解説画面。共有コンポーネント`ResultContent`）:
  符の内訳・成立役・計算式を1つのカードにまとめて表示する。上から順に:
  0. **見出し行**: 「内訳」ラベルの右に、正誤（`○ 正解`／`✕ 不正解`）と正解の点数
     （`答え: 8000`）を**同じ行に両端揃え**で表示する（左=「内訳」ラベル、右=正誤＋答え）。
     不正解時も**常に正解の点数**を表示する（ユーザーが選んだ値ではない）。マーク（○/✕）・
     「答え: 」の表記は他画面（符計算モードの解説等）と共通の規則を踏襲する。
     `QuizPage`のインライン結果（`collapsible`）では、この見出し行は「解説はこちら」トグルの
     **開閉状態に関わらず常時表示**する（畳んでいても正誤・答えだけは見える）。トグルで開閉する
     のは下記1〜3（符の内訳・成立役・計算式）と、カード外にある「高点法の別解」のみ。
  1. **符の内訳**: 符計算の各要素（基本符〔副底〕・門前ロン加符・ツモ符・面子の符・雀頭の符・待ちの符）を1行ずつ列挙する。同種の面子符は `暗刻(中張) × 2` のように集約し、符は合計値で表示する。待ちの形にはふりがなを添える（例: `嵌張(カンチャン)`）が、+0（両面・双碰）の待ちは表示しない。喰い平和形などの特殊な切り上げはその旨を注記する。最終符は計算式に含めて示すため、符の小計・合計行は表示しない。満貫以上（符が点数に影響しない区分）では符の内訳を表示しない。
  2. **成立役の一覧**（役名＋翻）。
  3. **計算式**（例: `40符3翻 (子ロン) → 5200`。親/子・ツモ/ロンを括弧内に示し、右寄せで表示）。満貫以上は符に代えて区分名を示す（例: `5翻 満貫 (親ロン) → 12000`）。
- **高点法の別解**: 他の解釈が存在する場合、別解とその点差を提示（「なぜ高い方を採るか」）。
- **符計算モードの解説**（`/fu/result`）は符の習得に主眼を置くため、共有カードを次のように扱う:
  - 正誤（○×）と正解の符は、点数計算モードとは異なり**独立した判定カード**（下記の旧デザイン）
    のまま表示する（見出し行への統合は行わない。点数計算モードのみの変更）。
  - **成立役の一覧は表示しない**。
  - **符の内訳を主役**とし、役名相当の強調スタイル（大きめ・太字・本文色）で表示する。
  - **+0符の要素も提示**する: 既定では省略する +0符 の要素（非役牌の雀頭・両面/双碰の待ち・**鳴き手のロン**）も含めて1行ずつ並べる。ただし**順子（0符）は列挙しない**。固定符（平和・七対子）は性質どおり1行表示を維持し、分解しない。0符要素は符の合計を変えない。
  - **面子の符ラベル（明刻/暗刻/明槓/暗槓）は色ドットで識別**する: 漢字が似て見分けにくいため、文字自体は他の行と同じ表示のまま、先頭に小さな色ドットのアクセントを添える（色相=刻子(緑)／槓子(琥珀)、塗り●=暗／枠○=明）。最終点数モードの内訳（補助情報）ではドットを付けず素のテキストのままにする。
- **正誤表示の見た目（点数計算モード）**: 点数計算モード（`QuizPage`・インライン結果／`ResultPage`解説画面）では、正解=ティール系／不正解=コーラル系の色付き枠＋グロー影＋登場アニメーションを持つ独立した「判定カード」は**廃止**し、上記「見出し行」内のテキスト表示に統合する。正誤の区別は「内訳」カード自体の枠・背景色ではなく、正誤の文字色のみで行う（正解=`--color-success-text`／不正解=`--color-danger-text`。既存の`.result-verdict.correct`/`.incorrect`と同じトークン）。答えの数値部分は数字専用フォント・`tabular-nums`・太字を維持しつつ、見出し行にふさわしいサイズへ縮小する。
  符計算モードの解説画面（`/fu/result`。`FuResultPage`は`ResultContent`を使わない独立実装）は、
  従来どおり独立した判定カード（正解=ティール系／不正解=コーラル系の色付き枠＋グロー影＋登場
  アニメーション、ゴールドピルの答え表示）のまま変更しない。点数換算モード（`/convert`）も同様に
  判定カードのまま変更しない。

### 4.5 成績・苦手復習

- **成績**を localStorage に保存: 累計正答率・履歴・現在の連続正解・最高連続正解。
- **苦手復習**: ミスを **符の型** と **役の種類** の2軸で記録し、正答率の低いタグの問題を重み付けして出題する。
- 成績画面で正答率・連続正解・苦手分野を可視化する。

### 4.6 共通フッター・プライバシーポリシー

- **全画面共通のフッター**を表示し、`プライバシーポリシー`・`運営者情報`（§4.11）へのリンクをどの画面からも提供する（広告掲載に伴う「容易に到達可能」要件を満たすため）。フッターは常に画面下端に配置する。

### 4.7 コンテンツ記事（学習ガイド）

- クイズ機能とは独立した**記事セクション**を持つ。目的は SEO・収益化（AdSense）で、点数計算の理解を助ける読み物コンテンツを提供する。
- 記事一覧（`/articles`）と記事詳細（`/articles/:slug`）の2画面を持つ。詳細URLは記事の `slug` に対応する（例: `/articles/tensu-keisan-guide`）。存在しない `slug` は一覧へフォールバックする。
- 記事本文は **Markdown** で管理し（`src/content/articles/`）、原稿ファイルをほぼそのまま描画する。HTML の直接埋め込みは無効化する（サニタイズ、XSS回避）。
- 記事内の**画像はプレースホルダ枠＋キャプション**で表示する（実画像は将来差し替え）。
- 記事内に**牌姿（手牌図）と役の翻数バッジを埋め込める**。クイズ画面と同じ牌画像コンポーネント（`TileFace`/`TileRow`）を再利用し、Markdown のフェンス付きコードブロック（言語識別子 `mahjong`）に、牌姿と翻数を**キー行**で書く（例: `hand: 45678p 123s 234m 99m | 9p` / `menzen: 1` / `naki: -`）。`hand` は `parseTileNotation` 準拠のグループ記法（数字＋末尾サフィックス `mpsz`、`0`＝赤5、`|` の右は上がり牌）。`menzen`／`naki` は**門前**／**鳴き（喰い下がり）**の翻数で、`-`（や `×`）は不成立（門前のみ等）を表す。描画は上部に**門前／鳴きの翻数バッジ**（鳴き不可はグレーの「なし」）を置き、その下に牌画像を横並びで出す。**あがり牌も手牌と区別せず、同じ大きさ・区切りなしで一列に並べる**（`|` はデータ上の区別としてのみ残し、区切り線・ラベル・枠強調は付けない）。**牌は本文幅に 13 枚＋あがり牌が収まるよう流動的に縮小**し、牌の周囲・囲いの左右余白は最小限にして牌を主役にする（スマホでも見切れ・横スクロールなく1画面に収める）。`hand:` キーの無い単一行は牌のみの記法として従来どおり解釈する（後方互換）。**CSS 背景画像ではなく代替テキスト付きの実 `<img>`** で表示し、牌アセットをクイズと二重管理しない（HTML の直接埋め込みの無効化は維持し、独自記法のみを描画対象にする）。不正な記法は素のコードブロックにフォールバックし描画を壊さない。
- 記事内の**リンクは出し分ける**: サイト内のツール／クイズへのCTAリンクは実際に遷移する `<Link>` として生かす。**未作成の記事**へのリンクはプレーンテキスト化し、404 に遷移しないようにする。外部リンクは新規タブで開く。
- 各記事は `document.title` と `meta description` を設定する。
- **回遊導線**: 全画面共通フッターに記事一覧への導線を持つ。ホームにも記事セクションへの導線を1つ置く。記事末尾からクイズ／ホームに戻れるCTAを配置する。
- **役一覧記事（`/articles/mahjong-yaku-list`）の掲載範囲**: 掲載対象は §5.1 の対象役リストのうち**手牌の形で決まる役**とする。翻数グループ（1翻／2翻／3翻／6翻／役満）ごとに、各役の**牌姿（`mahjong` ブロック）＋門前／鳴きの翻数バッジ＋「成立条件」「よくある勘違い」**を簡潔に記す（具体例節は設けず、1文を短く箇条書き主体）。
  - **除外する役**: **状況で決まる役**（リーチ・門前清自摸和〔ツモ〕・一発・海底/河底・嶺上・槍槓・ダブルリーチ等）は牌姿で表現できないため載せない（一発以降は元々 §2.2 のスコープ外）。ドラ・裏ドラ・赤ドラは役ではないため載せない。
  - **牌姿を省く役**: **三槓子・四槓子**は槓子を13枚＋あがり牌の記法で正しく表現できない（槓子は4枚で幅も超える）ため、`mahjong` ブロックを付けず**本文（成立条件・よくある勘違い）だけ**で説明する。翻数バッジも省く。

### 4.8 設定

- マイページ等が無いため、**単一の設定画面**（`/settings`）を用意する。導線は**ホーム**に置く（フッターには置かない）。
- 現時点の設定項目は**「切り上げ満貫」トグル（ON/OFF、既定ON）**のみ。子7700→8000／親11600→12000／子ツモ2000-3900→2000-4000／親ツモ3900オール→4000オールに切り上げるハウスルールで、§2.2 の通り任意設定として提供する。
- **点数計算モードの採点に反映**する: 設定ONの間、点数計算モードの出題・正解・4択・解説の点数はすべて切り上げ満貫ルールで計算される（§5.3の満貫しきい値を基本点1920に緩和）。**符計算モードには反映しない**（満貫以上を出題しないため無関係）。問題の生成・バンク自体は標準ルールのまま保持し、表示時に設定を反映する。
- **点数換算モードの採点にも反映**する（§4.9）: 出題セルの絞り込み（満貫未満の判定）と正解・4択の計算の両方に設定を用いる。設定ON時は `30符4翻`・`60符3翻` が満貫化するため出題対象から外れる。
- 設定ON時、点数計算モードの出題画面の**局条件タグの先頭**に「満貫切上」タグを表示する（§8.1）。既定ONのため、通常は最初から表示される。
- 永続化先は **IndexedDB**（端末内・外部送信なし）。読み込み不可・未保存時は既定値（**切り上げ満貫ON**）にフォールバックする。
- 将来的に IndexedDB から他の永続化手段（サーバDB等）へ移行する可能性があるため、実装は永続層を抽象化し、置き換えが局所的な変更で済む構成にする（詳細は `TASKS.md` P13）。

### 4.9 点数換算モード

**目的**: 符・翻から点数を即答する「点数早見表」の暗記・反復に特化したドリル。手牌の読み取り（和了分解・役判定・符計算）を切り離し、`符 × 2^(2+翻) → 支払い` の**換算だけ**を訓練する。

**位置づけ**:
- 手牌・ドラ・役・高点法・問題バンク・手牌ジェネレータは**一切用いない**。エンジンの `calculatePayment(han, fu, isDealer, winType)`（§5.3）と誤答生成 `generateChoices`（§7）のみを再利用する純粋な出題。
- 名称の近い「点数計算（最終点数）モード」とは別物で、**手牌を表示しない**点が最大の違い。

**出題**:
- お題は **符・翻・親子・ツモ/ロン** の4条件。4条件すべてを毎回ランダムに提示する。
- 出題する組み合わせは、**その時点の切り上げ満貫設定（§4.8）で満貫未満**（`determineRank(han, fu, { roundUpMangan }) === null`）かつ**実在し得る**セルに限定する（下表）。満貫以上は符が点数に影響しないため出題しない（§4.0）。
- **実在性の制約**（誤った暗記を避けるため）:
  - `20符` は**ツモのみ**・**2翻以上**（平和ツモ形。20符ロンは存在しない）。
  - `25符` は**2翻以上**（七対子）。
  - `30符以上` は1翻から、ロン・ツモ両方。
- 親子は自由。満貫未満の判定は基本点（親子非依存）で決まるため、成立するセルは親/子いずれでも成立する。

出題対象セル（基準となる候補セル。R=ロン可／T=ツモ可）:

| 符  | 1翻 | 2翻 | 3翻 | 4翻 |
| --- | --- | --- | --- | --- |
| 20  | —   | T   | T   | T   |
| 25  | —   | R/T | R/T | R/T |
| 30  | R/T | R/T | R/T | R/T |
| 40  | R/T | R/T | R/T | —（満貫） |
| 50  | R/T | R/T | R/T | —（満貫） |
| 60  | R/T | R/T | R/T | —（満貫） |
| 70  | R/T | R/T | —（満貫） | —（満貫） |

> 初期実装は 20〜70符に限定する。将来 80〜110符（暗槓を含む高符手）を追加してもよい。

**お題の表示（条件バッジ＋フラッシュカード）**:
- お題は上下2段で構成する。**上段に条件バッジ列**、**下段にフラッシュカード**を置く。手牌・牌画像は用いず、数字・文字のみで構成する。
- **上段（条件バッジ列）**: **親/子**・**ツモ/ロン**をバッジで表示する（従来どおり）。**満貫切上**（設定ON時のみ）も同列に含める（§4.8）。
- **下段（フラッシュカード）**: **符・翻**を1枚の出題カードとして、上部に最強調（最大のタイポ）で大きく表示する（例: `30符1翻`）。暗記カード＝フラッシュカードのメタファ。
  - 「何点？」等のプロンプトラベルは**置かない**。
- カードはスマホ最優先で中央寄せ。既存のデザイン基調（丸み・淡いグロー影のカード）と一貫させる。
- **回答結果はこの出題カードの中**（符・翻の下）に表示する（別カード・別画面にはしない。§4.9「採点・解説」）。
- カードは**回答結果分の領域を常に確保**し、**回答前後で高さが変わらない**ようにする。これにより下段のフラッシュカードより後ろに並ぶ**選択肢ボタン・「次の問題へ」ボタンの位置が動かない**。

**切り上げ満貫（§4.8）との反映**: 点数換算モードは**点数計算モードと同じ設定を参照**し、採点・出題セルの絞り込みの両方に反映する（点数計算モードとの数字の食い違いを避けるため）。
- 設定**OFF**（標準ルール）: 上表の通り。`30符4翻`・`60符3翻`（基本点1920）は満貫未満として出題対象（正解は `7700` 系）。
- 設定**ON**（切り上げ満貫）: 満貫のしきい値が基本点1920に緩和されるため、`30符4翻`・`60符3翻` は満貫化し**出題対象から除外**される。それ以外のセルは正解が切り上げ後の値（例: 親ロン以外は変化なし、境界セルのみ非該当）。
- 正解は `calculatePayment(han, fu, isDealer, winType, { roundUpMangan })`（§5.3）で算出する。

**回答UI**:
- 回答は**最終点数**を4択ボタンから選ぶ。選択肢形式は最終点数モードと同一で、親/子・ツモ/ロンにより切り替える（ロン=単一値／子ツモ=`X / Y`／親ツモ=`X オール`。§4.2）。
- 誤答は `generateChoices`（§7）を流用する（符切り上げ忘れ・翻±1/±2・親子取り違え・ツモ/ロン取り違え等。**実在点数のみ・正解と同一形式・4択**）。

**採点・解説（出題カード内インライン）**:
- 別画面へ遷移せず、選択直後に**出題カードの中**（符・翻の下）で正誤（○×）・正解の点数・**計算式**を表示する（符・翻は表示したまま。早見反復のため画面遷移を最小化）。
- 計算式は**簡潔な1本の等式**で示す。左辺＝符・累乗・倍率、右辺＝結果（必要な時だけ丸め前→丸め後の矢印を挟む）。
  - **指数部 `(2+N翻)` は上付き**（`2` の右肩に小さく `(2+N翻)`。累乗と分かるように）。
  - **ロン・親ツモ**（倍率が1つ）: 倍率を指数の直後に `×M(ラベル)` で置き、右辺は積そのもの。切り上げが生じる時だけ `積 → 点数` と矢印を足す。
    - 例: `50符 × 2^(2+1翻) ×4(子ロン) = 1600`
    - 例（切り上げあり）: `30符 × 2^(2+1翻) ×4(子ロン) = 960 → 1000`
  - **子ツモ**（子×1／親×2の2支払い）: 指数直後は `(子ツモ)` のみ。右辺は基本点→最終の「子 / 親」表示（既存の点数表記と同じ形。分数`1/2`は使わない）。
    - 例: `50符 × 2^(2+2翻)(子ツモ) = 800 → 800 / 1600`
- **レイアウト固定**: 出題カードは結果表示分の領域を常に確保し、回答前後で**選択肢ボタン・「次の問題へ」ボタンの位置が変わらない**ようにする。
- 「次の問題へ」で新しいお題を出し、カード内の結果表示をクリアする。

**成績**: 記録しない（§4.5 の集計に連携しない）。

**画面**: `/convert`（単一画面。出題・回答・解説をインラインで完結。§8.1）。

### 4.10 符分解モード

**目的**: 符の合計を一発で当てる符計算モード（§5.6）よりも一段細かく、**符を構成要素ごとに分解して組み立てる**訓練。各要素（上がり方・面子の符・雀頭・待ち）に「何符付くか」を選ばせ、初学者が符計算の内訳を体得できるようにする。

**位置づけ**:
- 手牌生成・符計算ロジックは符計算モードと共通（`nextProblem({ excludeMangan: true })` ＋ §5.2 のエンジン）。**回答方法だけが異なる**（合計符の4択 → 要素ごとの選択）。
- ドラは符に影響しないため**ドラ表示牌・裏ドラ表示牌は出さない**（符計算モードと同様）。
- **リーチも符に無関係のため表示しない**（局条件の「リーチ」バッジを出さない。ドラ非表示と同じく符ドリルのノイズを避ける）。リーチ手自体は出題され得る（プールは絞らない）が、リーチ表示のみを省く。
- **親/子（親子）も符に無関係のため表示しない**（局条件の「親/子」バッジを出さない）。符は親子に依存せず同一のため、リーチ表示を省くのと同じ理由で符ドリルのノイズを避ける。自風・場風は雀頭・面子の役牌判定に効くため表示は残す（自風=東 ⟺ 親 の含意で親子は暗黙に残る）。
- 満貫以上（`rank` 付き）は符が点数に影響しないため出題しない（§4.0・符計算モードと同一制約）。
- **七対子は約3%に抑える**: 符分解モードでは七対子は固定符（25符を `{20,25,30}` から選ぶだけ）で要素分解の練習にならないため、他モードよりも強く抑制し、実出題で**約3%**にとどめる（符分解モードは満貫以上を除外する結果、抑制しないと七対子の相対比率が約8〜9%まで上がる）。生成器の七対子生成率・問題バンクは変えず、**出題選択時のカテゴリ係数（§4.5 の七対子抑制係数）を符分解モード専用にさらに小さくして実現**する。最終点数・符計算モードの七対子頻度は変えない。
- 局条件（場風・自風）・手牌・上がり牌・ツモ/ロンは §4.1 に準拠して表示する（リーチ・親子を除く）。

**要素モデル（1画面フォーム一括回答）**:
お題の下に、符の各要素を縦に並べた**1枚のフォーム**を置き、各行で符を選ぶ。全要素を選んでから「採点する」で**一括採点**する（1要素ずつのウィザードにはしない）。手牌の形により2通り:

- **A. 標準手**（`FuBreakdown.fixed === false`。平和ロン・喰い平和形を含む）: 副底20は**採点対象外**（選ばせない）。画面上に独立した「副底20符」表示は置かず、採点後の合計式（`20 + …`）の中で示す。その上で、次の4要素を問う。
  1. **上がり方**: 門前ロン＝**10** / ツモ＝**2** / 鳴きロン＝**0**（平和ロンの門前ロン加符もここに含める）。選択肢 `{0, 2, 10}`。
  2. **面子の符（合計）**: 4面子の符の合計（Σ 面子符。§5.2 の面子符表）。個別ではなく**合計を1問**で問う。選択肢は正解を含む妥当な合計符プールの4択。**選択肢は昇順に表示**する（誤答の選定はランダムだが並びは値の小さい順に固定し、他の要素行＝固定選択肢が昇順なのと表示を統一する）。
  3. **雀頭**: 役牌（三元牌／自風／場風）＝**2** / それ以外＝**0**。選択肢 `{0, 2}`。
  4. **待ち**: 嵌張・辺張・単騎＝**2** / 両面・双碰＝**0**。選択肢 `{0, 2}`。
- **B. 固定符手**（`FuBreakdown.fixed === true`。平和ツモ20・七対子25）: 要素分解に馴染まないため、**固定符**の1要素のみを問う。
  1. **固定符**: 平和ツモ＝**20** / 七対子＝**25**。選択肢 `{20, 25, 30}`（混同しやすい値を誤答に充てる）。

> 平和ロン（30符＝副底20＋門前ロン10）は `fixed === false` のため**標準手（A）として分解**する（上がり方＝10・他0）。固定符（B）扱いになるのは `fixed === true` の平和ツモ20・七対子25のみ。

**正解の要素符**は §5.2 の符計算に一致する。エンジンの `calculateFuBreakdown` の内訳から、**文字列ラベルに依存せず要素カテゴリで**各要素の符を取り出す（実装は `TASKS.md` P16）。

**採点・解説（同一画面インライン）**:
- 「採点する」で各要素行に**○×**を表示する（要素ごとの正誤）。**○×は各要素名（ラベル）の右に置く**（例:「上がり方 ○」）。正解符自体は選択肢ボタンの配色（**自分が選んで正解＝緑／自分が選んで誤答＝赤／自分は選ばなかった正解値＝点線枠／それ以外の未選択肢＝グレーアウト**）で示し、正誤マークは○×のみとする。
  - **未選択肢のグレーアウト**: 採点後、各行で「**自分が選ばず、かつ正解でもない**」選択肢は**淡色（グレーアウト）**にして沈める。こうしないと未選択肢が既定表示（白地）のまま残り、自分が**何を選び／何を選ばなかったか**がパッと見で判別しにくいため。自分の選択（緑/赤）と正解値（点線枠）は沈めず、目立たせたままにする（正解値は学習のため常に提示する）。
- 合計と**10符切り上げ**は**自動表示**する（最終符自体は選ばせない）。標準手は `20 + Σ要素 = 合計 → 10符切り上げ = 最終符`、喰い平和形ロンは 20→30 の切り上げ注記（§5.2）も示す。固定符手は切り上げなし。
- 参考として §4.4 の符内訳（`FuBreakdownContent`、0符要素を含む）を下部に併記してよい。**成立役の一覧は表示しない**（符の習得に主眼／符計算モードと同じ方針）。
- **レイアウト固定**: 結果表示分の領域を常に確保し、「採点する」ボタンと下部のナビゲーションボタン（後述）の位置が回答前後で動かないようにする（§4.9 と同方針）。
- **要素行のラベル幅を揃える**: 標準手の4要素行（**上がり方・面子(合計)・雀頭・待ち**）は**すべて全幅で縦に1行ずつ**並べ、ラベル列の幅を **CSS Grid で共通固定**して、選択肢ボタンの開始位置（左端）を4行とも縦に整列させる。ラベルの文字数差でボタン列がずれないようにする。（旧仕様の「雀頭・待ちを横1行に並べる2択並列レイアウト」は廃止し、4要素を同一のラベル列幅・行レイアウトに統一する。○× を各要素名の右へ移したことで行上部の正誤マーク専用行が不要になり、縦の行数はむしろ減る。）
- **スマホ1画面表示**: スマホ縦画面では、手牌・お題（局条件・アガリ牌）・要素フォーム・採点結果を**スクロールせず1画面に収める**ことを目標とする。お題（局条件＋アガリ牌＋手牌）は上部にコンパクトに固定（sticky）し、選択肢を操作しても手牌が視界から外れないようにする。アガリ牌は独立ブロックにせず局条件列の隣にチップ配置する。参考の符内訳（下記）は既定で折りたたみ、1画面レイアウトを押し下げない。極小端末では手牌の常時可視（sticky）を優先し、最小限のスクロールに留める。
- **ナビゲーションボタン（←戻る／もう一度／次へ→）**: 採点結果の下に、出題を移動する3ボタンを**横一列**に並べる。
  - **←戻る**: 直前に出題した問題へ戻る（出題履歴を保持する）。履歴の先頭（最初の問題）では無効化する。
  - **もう一度**: 採点後に、**同じ問題**をもう一度解き直す（回答と採点結果をリセットし、問題は変えない）。採点前は無効化する。
  - **次へ→**: 次の問題へ進む（履歴の末尾にいれば新規出題、履歴の途中にいれば履歴上の次へ）。フォームと結果表示をリセットする。
  - 3ボタンは常時表示し、無効時も配置を保って**回答前後・採点前後でレイアウトが動かない**ようにする（§4.9 と同方針）。

**成績（非連携）**: 符分解モードは**成績に連携しない**（§4.5 の集計対象外。点数換算モード §4.9 と同じ扱い）。回答を `recordAnswer` で記録せず、上部ヘッダーに**成績リンクを表示しない**（導線はホームのみ）。全要素正解を「**完答**」と表示する採点表示自体は行うが、記録・苦手復習への反映はしない。出題の重み付け（§4.5 の苦手復習）は他モードで蓄積した成績を**読むだけ**で、符分解の回答は書き込まない。

**画面**: `/fu/parts`（単一画面。出題・要素回答・採点・解説をインラインで完結。§8.1）。

### 4.12 点数早見表ダイアログ

**目的**: 符・翻・親子・ツモ/ロンから最終点数を即引きできる「点数早見表」を、クイズの最中にいつでも参照できるモーダルとして提供する。反復学習で点数テーブルを暗記する助けとする。

**位置づけ**:
- **最終点数モード（`/quiz`）と点数換算モード（`/convert`）のヘッダー**（ハンバーガーの隣、`点数早見表を開く` ボタン）から開く共有部品（`ScoreTableDialog`）。ネイティブ `<dialog>` をモーダル表示する。
- セルの数値は**静的に持たず、エンジンの `calculatePayment`（§5.3）から生成**する。これによりクイズの採点結果と早見表が食い違わない（誤答提示の防止）。表示は**標準ルール（切り上げ満貫なし）**とする。

**内容**:
- **符×翻の本表**: 行＝符（20・25・30・40・50・60・70符）、列＝翻（1〜4翻）。各セルは上段にロン点、下段にツモ点（親＝各家オール額／子＝「子払い / 親払い」）を示す。`20符1翻`・`25符1翻` は実在しないため空セル。満貫以上のセルはアクセントで区別する。
- **満貫以上の区分行**: 符に依存しない区分（跳満6-7翻／倍満8-10翻／三倍満11-12翻／役満13翻~）を本表の下に続けて示す。

**親／子カルーセル**:
- 親の表・子の表を**2枚のスライド**として左右に並べ、**スワイプで切り替えられる**カルーセルにする。実装は **CSS `scroll-snap`**（横スクロールコンテナ＋`scroll-snap-type: x mandatory`、各パネル100%幅、スナップ）とし、タッチの慣性・スナップをネイティブに得る。**両パネルは常時DOMに描画**する（scroll-snap成立の前提）。
- ヘッダーの**親/子トグル**（セグメントコントロール）は残し、**スワイプと双方向同期**する。トグル押下でその表へ `scrollTo`（`prefers-reduced-motion` 時は即時ジャンプ）、スワイプで表が変わればトグルの選択状態（`aria-pressed`）も追従する（位置→state同期）。デスクトップ・非タッチ環境でもトグルで切り替えられる。
- **トグルの選択強調**: 形状（ピル型セグメントコントロール）は維持しつつ、選択中/非選択の対比を強める。
  - **選択中**: 現状どおり（teal塗り `#2ba8a2`・白文字・グロー影・`font-weight: 800`）。変更しない。
  - **非選択**: 文字色を `var(--color-text-sub)`（`#6a6a6a`、既存の共通ミュートトークン）にし、背景に薄いグレーを敷く（コンテナの薄teal地に対しても濁らないよう白ベースで合成した不透明色。透過teal色ではない）。`font-weight` は `600` に下げ、選択中の `800` と差をつける。
  - 非選択teal系文字色のままだと選択中と紛らわしく見える、という不具合の是正が目的（アクティブ側は変更しないため、この2点の変更だけで対比を確保する）。
  - hover挙動（非選択ボタンhover時のteal薄染まり `color-mix(in srgb, #2ba8a2 16%, transparent)`）は変更しない。
- **横スクロールはカルーセルの1軸のみ**とする。表自体は `table-layout: fixed` でパネル幅に収め、**表内の横スクロールは持たない**（横方向の入れ子スクロールによるスワイプ競合を避ける）。極小端末で数字が詰まる場合はフォント縮小で吸収する。**ダイアログ本文の縦スクロール**（区分行まで含む縦長表示）は残す。

**閉じる操作**: ×ボタン／背景（`::backdrop`）クリック／ESC のいずれでも閉じる（ネイティブ `<dialog>` の標準機能）。

**アクセシビリティ（§9）**: トグルは `aria-pressed`、スクロールコンテナは `aria-label` を持つ。各パネルに `親`／`子` の見出しを付す。色のみに依存しない情報提示とする。

### 5.3 点数計算

**基本点** = 符 × 2^(2 + 翻)

支払い（各支払いは**100点単位に切り上げ**）:

|      | 子（非親）                       | 親                |
| ---- | -------------------------------- | ----------------- |
| ロン | 基本点 × 4                       | 基本点 × 6        |
| ツモ | 子から 基本点×1、親から 基本点×2 | 各家から 基本点×2 |

**満貫以上（基本点に上限を設定）**

| 区分   | 適用翻                        | 子（ロン / ツモ）   | 親（ロン / ツモ）   |
| ------ | ----------------------------- | ------------------- | ------------------- |
| 満貫   | 4翻40符以上・3翻70符以上・5翻 | 8000 / 2000・4000   | 12000 / 4000オール  |
| 跳満   | 6〜7翻                        | 12000 / 3000・6000  | 18000 / 6000オール  |
| 倍満   | 8〜10翻                       | 16000 / 4000・8000  | 24000 / 8000オール  |
| 三倍満 | 11〜12翻                      | 24000 / 6000・12000 | 36000 / 12000オール |
| 役満   | 役満役・数え役満(13翻+)       | 32000 / 8000・16000 | 48000 / 16000オール |

**代表的な点数（満貫未満、参考）**

- 子ロン: 30符1翻1000／2翻2000／3翻3900／4翻7700、40符2翻2600／3翻5200、25符2翻1600／3翻3200／4翻6400
- 親ロン: 30符1翻1500／2翻2900／3翻5800、40符3翻7700
- （その他はすべて基本点の式から算出）

### 5.4 ドラの数え方

- **表示牌の枚数**（実戦準拠）: ドラ表示牌の枚数は **「1 ＋ 手牌中の槓（明槓・暗槓）の数」** とする（基本の1枚＋槓ごとに1枚めくられる「槓ドラ表示牌」）。槓が無ければ1枚、三槓子なら4枚、四槓子なら5枚（**上限5枚**）。麻雀ルール上「ドラ表示牌なし」はあり得ない。
- **裏ドラ表示牌の枚数**: リーチ時のみ提示し、その枚数も**表ドラ表示牌と同数（1＋槓の数）**。非リーチ時は0枚。
- 数え役満はドラ表示牌の積み上げではなく役で作る。
- **ドラ表示牌→ドラの変換ルール**（実戦準拠）:
  - 数牌（萬子/筒子/索子）: 表示牌の次の数字がドラ（例: 表示牌4索 → ドラは5索）。9の次は1に戻る（例: 表示牌9萬 → ドラは1萬）。
  - 風牌（字牌）: 東→南→西→北→東 の周期でドラが決まる（例: 表示牌東 → ドラは南）。
  - 三元牌（字牌）: 白→發→中→白 の周期でドラが決まる（例: 表示牌白 → ドラは發）。
- **ドラ**: 上記変換で得た実際のドラ牌が手牌（副露含む）に含まれる枚数だけ翻を加算。
- **裏ドラ**: リーチ時のみ。裏ドラ表示牌を同じ変換ルールで読み替え、手牌に含まれる枚数だけ加算。
- **赤ドラ**: 手牌中の赤5（赤牌）の枚数だけ加算。各色5に1枚（計3枚）を想定。ドラ表示牌の読み替えとは無関係。

### 5.5 高点法

- 手牌が複数の面子構成・待ち解釈を持つ場合、成立役・翻・符をそれぞれ算出し、**最終点数が最大**になる解釈を正解とする。
- 解説では別解と点差を提示する。

### 5.6 符計算モードの正解定義

- 符計算モードの正解は、§5.2 の**10符切り上げ後の符**（`ScoreResult.fu`）とする。
- **高点法適用後**の解釈の符を採る（＝最終点数が最大になる解釈の符。必ずしも「符が最大の解釈」ではない）。
- 満貫以上（`rank` 付き）は符が点数に影響しないため、符計算モードでは出題しない（§4.0）。
- 特殊ケース: 平和ツモ=20符、七対子=25符（固定）、喰い平和形ロン=30符（切り上げ）等も §5.2 の通り扱う。

```ts
  };
  doraIndicators: Tile[]; // ドラ表示牌（枚数=1+槓の数。§5.4。実際のドラは変換ルールで導出）
  uraDoraIndicators: Tile[]; // 裏ドラ表示牌（リーチ時のみ、枚数=1+槓の数。非リーチは空配列）
  conditions: {
    seatWind: Wind; // 自風
    roundWind: Wind; // 場風（east | south のみ）
    isDealer: boolean; // 親=true
    riichi: boolean;
  };
  answer: ScoreResult; // §5に基づく正解（高点法適用後）
  tags: {
    fuType: number; // 20/25/30/40/50/60/70...
    yakuCategories: string[]; // 苦手復習用の役カテゴリ
  };
}

interface YakuResult {
  name: string;
  han: number;
}

type Payment =
  | { kind: "ron"; total: number } // ロン: 単一値
  | { kind: "tsumo-ko"; nonDealer: number; dealer: number } // 子ツモ: 子X/親Y
  | { kind: "tsumo-oya"; each: number }; // 親ツモ: Xオール

interface ScoreResult {
  yaku: YakuResult[];
  han: number;
  fu: number;
  payment: Payment;
  rank?: "mangan" | "haneman" | "baiman" | "sanbaiman" | "yakuman";
  interpretationNote?: string; // 高点法の別解説明
}
```

- 問題バンクは上記 `Problem` の配列を JSON で持つ。
- **牌の合法性**: 同一牌は**最大4枚**（§4.1）。手牌全体（`hand.concealed`＋`hand.melds`。`concealed` は上がり牌を含む完成形）に加え、`doraIndicators`・`uraDoraIndicators`（表示牌も実牌）も合算して4枚を超えないこと。バンク・ジェネレータともにこの制約を満たす問題のみを採用する。
- `tags` は苦手復習の集計キー。`fuType` は §5.2 の丸め後の符、`yakuCategories` は成立役の分類（例: `"平和"`, `"対々和"`, `"混一色"`, `"役満"` など）。

## 7. 誤答（distractor）生成方針

正解 `(han, fu, payment)` から、**ありがちな誤り**を再現した誤答を生成する:

- 符を**切り上げずに**計算した点数
- **翻±1**（役の数え間違い）
- **親／子を取り違え**た点数
- **ツモ／ロンを取り違え**た点数（同じ手の別支払い形式）
- **ドラ枚数±1**
- 一つ上／下の区分（満貫↔跳満 等）

生成した候補は、正解と同じ**支払い形式（親/子・ツモ/ロン）に揃え**、重複を排除し、正解を必ず含めて**4個**にした上でシャッフルする。値は実在し得る点数に丸める。

**符計算モードの誤答**は、正解の符内訳 `fuDetail`（`total`・`subtotal`・`fixed`）から、ありがちな符の誤りを再現する:

- **切り上げ忘れ**（`subtotal` そのまま。`total` と異なる場合）
- **1段階のずれ**（`total ± 10`）
- **待ち符・ツモ符の付け忘れ／数え過ぎ**（`total ± 2` の近似）
- 妥当な符プール（20/25/30/40/50/60/70）から補完
- 固定符（七対子25符・平和20符）は `±10` 系を避け、混同しやすい他符（20/30/40/50）を充てる

正解の符を必ず含め、重複を排除して4個にした上でシャッフルする。

### 8.4 ファビコン・アプリアイコン・マニフェスト

- **目的**: ブラウザタブ・ブックマーク・iOS/Android「ホーム画面に追加」時に表示されるアプリのアイコンと名称を整え、ブランドの一貫性と信頼性（AdSense 審査・共有時の見栄え）を高める。加えて、Web アプリマニフェストにより**ホーム画面へのインストール（スタンドアロン表示）**を可能にする（ただし Service Worker は持たず、オフラインでは動作しない。§2.2）。
- **配置**: アイコン資産一式は **`public/icons/` 配下**に置く（Vite が `public/` をルート配信し、ビルド時に `dist/` へそのままコピーするため追加のバンドル設定は不要）。参照は絶対パス（`/icons/favicon.ico` 等）とする。
- **提供する資産**:
  - **ファビコン**（`/icons/favicon.ico`）: ブラウザタブ／ブックマーク用。16・32px を含むマルチサイズ `.ico`。広い互換のため基本とする。
  - **SVG ファビコン**（`/icons/icon.svg`）: 対応ブラウザで高解像度表示に用いる。
  - **apple-touch-icon**（`/icons/apple-touch-icon.png`、180×180px）: iOS/iPadOS の「ホーム画面に追加」用。角丸・余白は OS 側が付与するため画像自体は角丸なし・全面塗り。
  - **マニフェストアイコン**（`/icons/icon-192.png`・`/icons/icon-512.png`。512 は `maskable` も兼ねる）: `site.webmanifest` から参照。
  - **Web アプリマニフェスト**（`/icons/site.webmanifest`）: `name`「麻雀点数計算」／`short_name`「点数計算」／`display: standalone`／`start_url: "/"`／`theme_color`・`background_color`＝`#146F45`。
- **HTML への反映**: `index.html` の `<head>` に以下を追加する。
  - `<link rel="icon" href="/icons/favicon.ico" sizes="any">`
  - `<link rel="icon" href="/icons/icon.svg" type="image/svg+xml">`
  - `<link rel="apple-touch-icon" href="/icons/apple-touch-icon.png">`
  - `<link rel="manifest" href="/icons/site.webmanifest">`（マニフェスト実体は `public/icons/` 配下のため、パスは `/icons/site.webmanifest` とする。付属スニペットの `/site.webmanifest` は誤りのため修正して反映する）
  - `<meta name="theme-color" content="#146F45">`
- **スコープ外**（本仕様では扱わない）: Service Worker・オフラインキャッシュ（§2.2）、OGP/Twitter カード用の `og:image` メタ。`public/icons/` 内の `head-snippet.html`・`preview.png`・`icon-1024.png` はソース／確認用の同梱物で、`<head>` からは参照しない（配信されても未使用）。

### 8.5 ヘッダーナビゲーション（ハンバーガー＋サイドバー）

- **目的**: ヘッダー内に露出していたナビゲーション（「ホーム」「成績」インラインリンク）を**ハンバーガーメニュー＋サイドバー（ドロワー）**に集約し、ヘッダーを簡潔にする。**再利用可能な共通部品**（ハンバーガーボタン・サイドバー）として実装する。
- **適用範囲**: 出題・解説系の6画面——**最終点数**（出題 `/quiz`・解説 `/result`）・**符計算**（出題 `/fu/quiz`・解説 `/fu/result`）・**符分解**（`/fu/parts`）・**点数換算**（`/convert`）。他画面（成績・記事・設定・About 等）は**現状の `PageHeader`（インラインの「ホーム」「成績」リンク）を維持**する（共通部品化しているため、後続フェーズで展開してもよい）。
- **ヘッダーの構成**: 帯スタイル（§8.3）は維持する。帯の中に**左＝見出しタイトル**（例:「点数計算」「解説」「符計算」「符分解」「点数換算」）、**右＝ハンバーガーボタン**を置く。右からのドロワーと開閉の動線を一貫させるためハンバーガーは右端に配置する。従来ヘッダーにあった「ホーム」「成績」インラインリンクは**対象画面からは撤去**する。
- **サイドバー（ドロワー）の挙動**:
  - ハンバーガーボタンを押すと、**画面右からドロワー**がスライドインし、**背景を半透明オーバーレイ**で覆う。
  - サイドバー内は上から**「ホーム」**（`/` へ遷移）→**モード切替ボタン**（後述）→成績に連携するモードのみ**「成績」**（`/stats` へ遷移）の順に縦に並べる。**成績に連携しないモード（符分解・点数換算）は「成績」を出さない**（§4.10・§4.9 の成績非連携方針に合わせる）。
  - **成績ボタンは既存の状態引き継ぎを維持**する: `backTo`（出題パス。最終点数は `/quiz`、符計算は `/fu/quiz`）と現在の `problem` を遷移 state として渡し、成績画面の「練習に戻る」で**同じ問題（同じ4択・同じ並び順）に戻れる**挙動を保つ（現行 `PageHeader` と同等）。
  - **閉じる操作**: ①オーバーレイ（背景）タップ／②サイドバー内の×（閉じる）ボタン／③ESC キー——のいずれでも閉じる。
- **モード切替ボタン**: 「ホーム」の直後に見出し「他のモードで練習」を置き、**現在表示中のモードを除いた他モードへの遷移ボタン**（最大3つ）を並べる。
  - **モードは4つ**: 点数計算モード（`/quiz`）・符計算モード（`/fu/quiz`）・符分解モード（`/fu/parts`）・点数換算モード（`/convert`）。
  - **判定はモードファミリー単位**: 出題画面と解説画面は同一モードとして扱う。`/result` は「点数計算モード」、`/fu/result` は「符計算モード」として判定し、それぞれ自モードのボタンは出さない。
  - **現在のモードはボタンごと非表示**にする（無効化表示にはしない）。例: 符分解モード（`/fu/parts`）中は「点数計算モード」「符計算モード」「点数換算モード」の3つのみ表示。
  - 各ボタンは対応モードの出題画面（点数計算→`/quiz`、符計算→`/fu/quiz`、符分解→`/fu/parts`、点数換算→`/convert`）への**通常のリンク**。確認ダイアログなし・state 受け渡しなしで**即座に遷移**する（未回答の問題を離脱しても成績への影響はないため。§4.5）。
  - **`currentMode` は呼び出し元ページが明示的に `SidebarPageHeader` へ渡す**（`backTo`・`showStats` と同様、呼び出し元が明示する設計に統一。URL パターンからの自動判定は行わない）。
  - **モード定義の共有**: モードの id・表示ラベル・遷移先パス・アイコン（絵文字）は共有の定義（`src/config/modes.ts`）に集約し、ホーム画面（§8.1 画面一覧 1.）のモード選択カードと本機能の両方から参照する（表示名・パスの二重管理を避ける）。ホーム画面固有の説明文・牌画像装飾・個別 CSS クラスは共有せず、ホーム画面側に残す。
  - **アイコン**: 既存の「ホーム」🏠・「成績」📊と同系統の絵文字を使う（点数計算🧮／符計算🔢／符分解🧩／点数換算📐）。
  - **適用範囲**: 本節冒頭の対象6画面（`/quiz`・`/result`・`/fu/quiz`・`/fu/result`・`/fu/parts`・`/convert`）すべて。
- **アクセシビリティ（§9）**: ハンバーガーボタンに `aria-expanded`／`aria-label` を付与する。開いたらサイドバーにフォーカスを移し、フォーカスをドロワー内にトラップする。色のみに依存しない情報提示とし、タップターゲットは十分な大きさを確保する（§8.2）。
- **実装上の注意**: サイドバーは `document.body` へ portal して描画する。対象画面のヘッダー帯（§8.3）は `backdrop-filter` を持つため、ヘッダーの子として描画すると `position: fixed` の包含ブロックがビューポートでなくヘッダー自身になり、ドロワー・オーバーレイが破綻する（CSS仕様上の挙動）。portal 化によりこの問題を回避している。

## 9. 非機能要件

- **対応ブラウザ**: モバイル/デスクトップの最新版 Chrome・Safari・Edge・Firefox。
- **パフォーマンス**: 出題→表示は体感即時（問題生成は事前/非同期でも可）。エンジンの1問採点は数ms〜十数ms目標。
- **正確性**: エンジンは §5 をユニットテストで網羅（問題バンクを正解データに使う回帰テスト）。
- **アクセシビリティ**: 牌画像に代替テキスト、色のみに依存しない情報提示、フォーカス可能なUI。
- **配信**: 静的ホスティング（GitHub Pages 等）。

## 10. 受け入れ基準（Acceptance）

- ランダム出題→回答→採点→解説→成績、の一連が動作する。
- 選択肢が親/子・ツモ/ロンで正しい形式に切り替わる。
- 高点法で正解が決定され、解説に内訳と別解が表示される。
- 苦手分野が符の型・役の種類の2軸で記録・可視化される。
- §2.2 のスコープ外要素が混入していない（状況役・ハウスルール等）。
- エンジンのユニットテストが全て通り、問題バンクの正解と一致する。
- **すべての出題でドラ表示牌が1枚以上表示される（「なし」が出ない）**。
- **符計算モードで出題→符回答→採点→符内訳付き解説、の一連が動作する**（満貫以上は出題されない）。
- **点数換算モードで、符・翻・親子・ツモ/ロンの条件から点数を4択回答→同一画面で正誤・正解点数・計算式表示、の一連が動作する**（満貫以上は出題されない・手牌は表示しない・成績には連携しない）。
- **符分解モードで、符の各要素（標準手＝上がり方・面子の符・雀頭・待ち／固定符手＝固定符）を選択→一括採点→要素ごとの○×・合計・10符切り上げ表示、の一連が動作する**（満貫以上は出題されない・ドラは表示しない・**成績には連携しない**）。「←戻る／もう一度／次へ→」で出題履歴を前後・再挑戦できる。
- **運営者情報ページ（`/about`）が表示され、全画面共通フッターの「運営者情報」リンクからどの画面でも到達できる**（サイト目的・運営者名義・提供コンテンツ・広告掲載の明記・免責・お問い合わせを記載）。
- **記事内の牌姿（`mahjong` コードブロック）がクイズと同じ牌画像で描画され、門前／鳴きの翻数バッジが表示される**（CSS 背景ではなく代替テキスト付き `<img>`。牌はスマホ本文幅に収まり見切れない）。不正な牌記法はコードブロックにフォールバックし描画が壊れない。**役一覧記事（`/articles/mahjong-yaku-list`）が、手牌の形で決まる各役（§5.1 のうち状況役を除く）の成立条件・よくある勘違い・門前／鳴きの翻数バッジつきで読める**（状況役〔リーチ・ツモ等〕は非掲載、三槓子・四槓子は牌姿を省き本文のみ）。
- **出題・解説系の6画面（`/quiz`・`/result`・`/fu/quiz`・`/fu/result`・`/fu/parts`・`/convert`）で、ヘッダーのハンバーガーボタンから右ドロワーのサイドバーが開き、「ホーム」（成績に連携するモードは「成績」も）へ遷移できる**（§8.5）。ヘッダーからインラインの「ホーム」「成績」リンクは撤去されている。サイドバーはオーバーレイタップ／×ボタン／ESC のいずれでも閉じ、成績ボタンからは「練習に戻る」で同じ問題（同じ4択・並び順）に戻れる（状態引き継ぎを維持）。符分解・点数換算モードのサイドバーには「成績」を出さない（成績非連携の方針を維持）。**他画面（`PageHeader` 利用）のヘッダーとインラインリンク挙動には回帰がない**。
- **ファビコン・apple-touch-icon・Web アプリマニフェストが反映されている**（§8.4）: ブラウザタブに `favicon.ico`／`icon.svg` が表示され、iOS「ホーム画面に追加」で `apple-touch-icon.png` が使われ、`site.webmanifest`（`/icons/site.webmanifest` を正しく参照）によりホーム画面インストール（スタンドアロン表示）が可能になる。資産は `public/icons/` に置かれビルド出力（`dist/`）へコピーされる。Service Worker・オフラインキャッシュは導入していない（オフラインでは動作しない）。
