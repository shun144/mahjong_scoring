# 出題ランダム性とテスト失敗の調査メモ

このメモは「バンク抽選を25%に変更したら、あるテストが失敗した」一件で何が起きたかを、
**(A) 専門用語ありの正確な説明** と **(B) 中学生でもわかる平易な説明** の2種類で残すもの。

---

## 経緯（共通の前提）

- やりたかったこと: 出題の内訳を「その場で作る問題（生成）75% ／ 用意済みの問題集（バンク）25%」に変える。
  - 変更前は 50% / 50% だった。コードは [nextProblem.ts](../src/store/nextProblem.ts) の `chance(0.75, rng)` の1文。
- 起きたこと: 変更後、「**苦手なカテゴリの問題が多く出題されるか**」を確かめるテストが失敗した。
- 結論: **機能（苦手復習）は正しく動いていた**。失敗したのは**テストのやり方が不安定**だったから。
  最終的にテストを安定な形に作り直して解決した。

---

# (A) 専門用語ありの説明

## 1. 何を確かめるテストだったか
「あるカテゴリ（例: 七対子）を何度も間違えると、そのカテゴリの問題が出やすくなる」という
**苦手復習（重み付け出題）**の検証テスト。[nextProblem.test.ts](../src/store/nextProblem.test.ts)。

## 2. なぜ失敗したか（表面）
テストは **シード付き乱数（seed）** を使って出題を再現し、「七対子の出題率」を実測して
苦手化の前後で比較していた。しかし今回の変更で、乱数を消費する順序・回数・分岐がずれ、
**実測値がわずかな差でひっくり返った**。当初は偶然の揺れ（サンプリング分散）を疑い、
複数シードでのプール集計・試行回数の増加で対処しようとした。

## 3. 真因（掘り下げ）
- まず重み付けロジック自体が正しいかを、乱数を使わず検証した。
  - 出題確率は `problemWeight(p, stats) × categoryBias(p)` に比例する（[weighting.ts](../src/store/weighting.ts)）。
  - **解析計算**（重み合計の比）と **モンテカルロ**（`weightedPick` を20万回）の双方で、
    七対子の出題確率は **0.021 → 0.030** に上がると確認。つまり機能は正しい。
- それでも `nextProblem` 経由の統合テストでは、同一の乱数列（同じ4976件のバンク抽選）なのに
  苦手化後の方が七対子を引かなかった。原因は **疑似乱数（PRNG）の系列相関**。
  - シード付き乱数は **Park–Miller 型の線形合同法（LCG）**（`state = state * 16807 % 2147483647`）。
    格子構造（lattice structure）があり、連続する値に相関がある。
  - `nextProblem` のバンク抽選に渡る乱数 `r` は「`chance` の抽選（`rng() ≥ 0.75`）の**直後**の乱数」
    という**部分列**。この部分列が一様分布にならず、`weightedPick` の結果を系統的に歪めていた。
  - 一方、私のモンテカルロは連続する乱数をそのまま使ったので歪まず、正しい 0.030 が出た。
- まとめると、**このテストは「低品質な乱数の癖 × 乱数消費の順序 × バンク配列の並び」に依存した脆いもの**
  だった。元々シード7の単一サンプル・僅差で、偶然通っていたにすぎない。
  （なお、`recordAnswer` が問題の全タグ（七対子だけでなく「リーチ」等）を苦手化する
  **交絡**も疑ったが、解析値が上昇することから主因ではないと確認した。）

## 4. どう直したか
乱数サンプリングによる統計比較をやめ、**`nextProblem` のバンク抽選が実際に使う重み
（`problemWeight × categoryBias`）で「七対子の出題確率」を決定論的に計算し、苦手化の前後で比較**する
形にテストを書き換えた。
- 長所: PRNG の系列相関やサンプリング分散に一切左右されない。高速（ミリ秒）で安定。
  機能の本質（重み付け）を直接検証している。

## 5. 教訓
- ランダム性を含む挙動を「実測サンプリング＋僅差の大小比較」で検証すると、
  乱数実装の癖や消費順序の変化で誤検知しやすい。
- 可能なら**期待値（確率・重み）を決定論的に検証**する。乱数サンプリングに頼るなら、
  効果量に対して十分なサンプルと、PRNG 品質への非依存を確保する。

# (B) 中学生でもわかる説明（専門用語なし）

## 1. このテストは何を見ていた？
このアプリには「**同じ種類の問題を何回も間違えると、その種類の問題が前より多く出る**」という
親切な仕組み（苦手を集中的に練習させる機能）がある。テストは「その仕組み、ちゃんと働く？」を
チェックしていた。

## 2. まず「ランダムなのに毎回同じ」の話
コンピュータの「ランダム」は、じつは**サイコロを本当に振っているわけではない**。
「ある数字から出発して、決まった計算で次々に数を作る」だけ。だから
**出発点の数字（合言葉のようなもの）を同じにすると、毎回まったく同じ順番で数が出てくる**。
テストでは、この“出発点”を固定して「毎回同じ配り方」にして、条件をそろえて比べていた。

## 3. なぜ失敗した？
今回、別の修正で「途中でくじを1回引くのをやめた」ため、**そこから先の数の出る順番が全部ズレた**。
テストは「たった1回試した結果」で、しかも比べている差が**ほんのわずか**だった（たとえば
750問中の20問と21問くらい）。だから順番が少しズレただけで、たまたま逆の結果になり、失敗した。

## 4. 調べてわかった本当の原因
- **機能そのものは正しく動いていた。** 「間違えた種類が出やすくなる」ことは、
  計算でもシミュレーションでも、ちゃんと確認できた（出やすさが上がっていた）。
- 失敗の原因は、**このアプリのランダムが“クセのあるランダム”だったこと**。
  本物のサイコロと違って、出る数字に少し規則性（前の数字の影響）が残っていた。
  そして、テストが**そのクセに引っかかる一部の数字だけを使っていた**。

### 具体例：「6の次だけ見る」いかさまサイコロ
かなり単純化すると、こういうこと。

アプリは1問出すのに、乱数（0〜1の数）を順番に使う。ざっくり2ステップ:

1. **1回目の数**で「新しく作る／問題集から選ぶ」を決める（数が大きい＝0.75以上なら問題集）。
2. 問題集を選んだときだけ、**2回目の数**で「問題集の中のどれにするか」を決める。

ここがポイント。「どの問題にするか」は、いつも
**“大きい数（0.75以上）の直後に出た数”**で決まる。全部の数を使うわけではなく、
**特定の場所（大きい数の次）の数だけ**を使っている。

いっぽう、このアプリのランダムには隠れた規則性があった。たとえるなら
**「6が出た次は、3が出にくい」いかさまサイコロ**のようなもの。
- 3は本来 1/6 くらいで出るはず。
- でも「**6の次の目だけ**」を集めて数えると、3がやけに少なく見える。
  3が本当に少ないのではなく、**“6の次”という偏った場所ばかり見ている**から。

アプリでも同じことが起きた:
- 「どの問題か」を決める数は、いつも「大きい数の直後の数」＝**偏った場所の数**ばかり。
- 「七対子」は、ちょうどその偏りのせいで**引かれにくい範囲**に入ってしまった。
- だから、七対子の出やすさを上げたのに、テストの数え方だと「**むしろ減った**」ように見えた。

要するにテストは、**サイコロ全体ではなく“6の次”だけを覗いていた**ので、
本当の出やすさとズレた数字を見てしまった、というわけ。
（証拠に、“全部の数”をそのまま使って数え直すと、七対子はちゃんと増えていた。）

- つまり、テストのやり方が**「運のよしあし」に左右される作り**になっていた。
  今まで通っていたのは、たまたま運がよかっただけ。

## 5. どう直した？
「実際に何回も出題してみて数える」というやり方（運に左右される）をやめて、
**「どの問題がどれだけ出やすいか」という“出やすさの点数”を直接計算して比べる**やり方に変えた。
- これなら、ランダムのクセや運に関係なく、**いつでも同じ・すぐに・正しく**判定できる。
- 結果として、「苦手にした種類は、ちゃんと出やすくなる」ことをきちんと確認できた。

## 6. ひとことで
**アプリの機能は正常。ただテストが「1回勝負・僅差・クセのあるランダム頼み」で不安定だったので、
運に左右されない“計算で確かめる”方法に変えて安定させた。**

## 付録: 用語の対応表（A↔B）

| 専門用語(A) | 平易な言い換え(B) |
|---|---|
| シード / PRNG（疑似乱数） | 出発点の数字 / クセのあるランダム |
| Park–Miller 型 LCG・系列相関 | 前の数字の影響が残る規則性 |
| サンプリング分散・僅差 | 運のブレ・ほんのわずかな差 |
| 重み（problemWeight × categoryBias） | 出やすさの点数 |
| 解析計算・モンテカルロ | 計算で確認・たくさん試して確認 |
| 決定論的テスト | 運に左右されない確かめ方 |
| 交絡 | ついでに別の種類まで苦手扱いされる紛れ |
